## 原因分析
- 对应路径 `/article/2018/12/02/编程技巧分享` 仍为 404 的主要原因不是路由参数，而是内容在 MDX 序列化阶段报错，导致 `getStaticProps` 捕获异常后返回 `{ notFound: true }`。
- 构建日志已出现多条 MDX 编译错误，包含 `Unexpected character '/'`、`'!'`、`'<'` 等，均会触发我们当前的 `catch` 分支，最终返回 404（见 `src/pages/article/[yyyy]/[mm]/[dd]/[title].tsx:204-207`）。
- 即使启用 `fallback: 'blocking'`，首次访问仍会调用 `getStaticProps`，一旦序列化报错，仍返回 404。

## 解决方案（不新增依赖的最小改动）
1. 在 `getStaticProps` 内为 MDX 序列化添加细粒度的 `try/catch`：
   - 如果 `serialize(article.content)` 抛错，优先用“代码块包裹”的方式降级渲染：将原始内容包裹为代码块字符串（如 ```\n<原文>\n```），再进行 `serialize`。
   - 这样可规避绝大多数因裸 HTML、斜杠、感叹号等导致的 MDX 解析错误，同时避免 404。
2. 调整最外层 `catch`：
   - 避免直接 `{ notFound: true }`，改为返回带“降级内容”的页面，页面上提示“内容暂时无法渲染为富文本”。
3. 保持现有 UTC 路径参数与 `fallback: 'blocking'`、`revalidate: 60` 设置不变。

## 代码修改点（伪代码）
- 文件：`src/pages/article/[yyyy]/[mm]/[dd]/[title].tsx`
  - 在找到文章后：
    ```pseudo
    let serializedContent
    try {
      serializedContent = await serialize(article.content || '')
    } catch (e) {
      const safe = '```\n' + (article.content || '') + '\n```'
      serializedContent = await serialize(safe)
    }
    return { props: { article: { ... , content: serializedContent } }, revalidate: 60 }
    ```
  - 在函数最外层 `catch`：
    ```pseudo
    const fallbackText = '内容暂时无法渲染为富文本。以下为原始文本：\n\n' + (raw || '')
    const content = await serialize('```\n' + fallbackText + '\n```')
    return { props: { article: { ... , content } }, revalidate: 60 }
    ```
